# Contributing to OTOI Framework

Thank you for your interest in contributing to the OTOI Framework! We welcome contributions from everyone, especially neurodivergent developers and accessibility advocates.

## 🎯 Our Mission

We're building user-centric AI interaction standards that put people in control of how AI systems work with them. This is especially important for neurodivergent users who need AI that supports rather than overwhelms their cognitive patterns.

## 🤝 How to Contribute

### 1. **Report Issues**
- Use our [issue templates](.github/ISSUE_TEMPLATE/) to report bugs, request features, or ask questions
- Be specific about accessibility needs or neurodivergent considerations
- Include examples and context when possible

### 2. **Suggest Features**
- We especially welcome suggestions for ADHD and neurodivergent support features
- Consider how your suggestion affects different cognitive patterns
- Think about accessibility and user agency implications

### 3. **Submit Code**
- Fork the repository and create a feature branch
- Follow our coding standards (see below)
- Test your changes thoroughly
- Submit a pull request with a clear description

### 4. **Improve Documentation**
- Help make technical concepts more accessible
- Add examples and visual aids where helpful
- Consider different learning styles and cognitive patterns

## 🧠 Neurodivergent-Friendly Development

### **ADHD Support**
- **Clear Structure**: Use descriptive names and logical organization
- **External Memory**: Comprehensive comments and documentation
- **Progress Tracking**: Clear milestones and completion indicators
- **Reduced Cognitive Load**: Minimize complexity and jargon

### **Autism Support**
- **Predictable Patterns**: Consistent coding styles and conventions
- **Clear Communication**: Explicit instructions and expectations
- **Sensory Considerations**: Avoid overwhelming visual or text density
- **Special Interest Integration**: Connect features to personal interests

### **General Neurodivergent Support**
- **Multiple Complexity Levels**: Basic → Intermediate → Advanced
- **Visual Organization**: Use diagrams, charts, and clear formatting
- **Flexible Workflows**: Support different working styles and preferences
- **Inclusive Language**: Respectful, clear, and jargon-free communication

## 📝 Coding Standards

### **Code Style**
- **Python**: Follow PEP 8 with 100-character line limit
- **JavaScript**: Use Prettier with 2-space indentation
- **Markdown**: Use consistent formatting and clear headings
- **JSON**: Use 2-space indentation with sorted keys

### **Documentation**
- **Comments**: Explain the "why" not just the "what"
- **README Files**: Include clear examples and use cases
- **API Docs**: Provide complete examples with expected outputs
- **Accessibility**: Consider screen readers and cognitive accessibility

### **Testing**
- **Unit Tests**: Test individual functions and components
- **Integration Tests**: Test complete workflows and user scenarios
- **Accessibility Tests**: Verify neurodivergent-friendly features
- **Performance Tests**: Ensure fast loading and efficient operation

## 🚀 Development Setup

### **Prerequisites**
- Python 3.8+ (for tools and validators)
- Node.js 16+ (for web components)
- Git (for version control)
- Your preferred code editor (we recommend Cursor!)

### **Quick Start**
```bash
# Clone the repository
git clone https://github.com/neurolift/nlt-otoi.git
cd nlt-otoi

# Install dependencies
pip install -r requirements.txt
npm install  # if working on web components

# Run tests
python -m pytest tests/
python tools/validators/schema-validators.py

# Start development server (if applicable)
npm run dev
```

### **Project Structure**
```
nlt-otoi/
├── docs/           # Documentation
├── schemas/        # JSON schemas
├── templates/      # User templates
├── examples/       # Implementation examples
├── tools/          # Development tools
├── tests/          # Test suites
└── .github/        # GitHub configuration
```

## 🔍 Review Process

### **Pull Request Guidelines**
1. **Clear Description**: Explain what you changed and why
2. **Accessibility Impact**: Describe how changes affect neurodivergent users
3. **Testing**: Include test results and validation output
4. **Documentation**: Update relevant documentation
5. **Examples**: Provide usage examples where applicable

### **Review Criteria**
- **Functionality**: Does it work as intended?
- **Accessibility**: Is it neurodivergent-friendly?
- **Documentation**: Is it well-documented and clear?
- **Testing**: Are there adequate tests?
- **Performance**: Does it meet performance requirements?

### **Review Timeline**
- **Initial Review**: Within 48 hours
- **Follow-up Reviews**: Within 24 hours of updates
- **Merge Decision**: Within 1 week for most changes

## 🎨 Design Principles

### **User Agency**
- Users should always have control over their AI interactions
- No hidden behaviors or unexpected changes
- Clear opt-in/opt-out mechanisms

### **Transparency**
- Open source and community-driven
- Clear documentation of all features
- Honest communication about limitations

### **Accessibility**
- Support for different cognitive patterns
- Multiple ways to access information
- Respect for individual preferences and needs

### **Privacy**
- User control over data use and retention
- Clear data handling policies
- No unnecessary data collection
